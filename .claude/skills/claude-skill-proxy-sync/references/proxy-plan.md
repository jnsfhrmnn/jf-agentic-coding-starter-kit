# Repository-Local Proxy Plan

## Goal

Let Claude and Codex use the same repository-local skills without duplicating
their implementation and without requiring user-global installation.

## Source Of Truth

The canonical skill is `.claude/skills/<name>/SKILL.md`. A Codex proxy contains
only trigger metadata and runtime instructions that load that canonical file.
Relative resources mentioned by the canonical skill resolve from its own
directory.

## Generated Layout

```text
.claude/skills/<name>/
  SKILL.md                 canonical behavior
  agents/openai.yaml       optional canonical UI metadata
  scripts/ references/     optional canonical resources

.codex/skills/<name>/
  SKILL.md                 generated thin proxy
  agents/openai.yaml       generated Codex UI metadata
```

The proxy marker stores only a repository-relative canonical path, a contract
hash, the source kind `repo`, and the schema version. It must never contain a
machine-specific absolute path.

## Freshness Model

The contract hash covers:

- frontmatter `name`;
- frontmatter `description`;
- `agents/openai.yaml` interface metadata.

Body, script, reference, and asset changes are deliberately excluded because the
proxy reads the canonical skill at runtime. This gives live body updates while
still detecting changes that Codex must index before activation.

## Conflict And Deletion Rules

- Generated targets may be updated deterministically.
- Existing targets without the generator marker are unmanaged conflicts and are
  never overwritten.
- Removed canonical skills become stale proxies.
- Stale proxies are removed only with explicit `--apply --prune`, and only if the
  marker proves they were generated by this system.

## Non-Goals

- No global Claude or Codex skill installation.
- No absolute canonical paths.
- No copying full Claude skill bodies into Codex.
- No cross-repository synchronization.
- No sync of Claude agents, general rules, settings, or user guidance.
